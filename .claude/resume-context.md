## 💼 项目经历

### 项目一：灵思港
**时间**: 2025.1 - 2025.3

#### 项目介绍
本项目是创意与灵感的停泊港湾，作为视频平台，功能涵盖内容上传、社区互动、个人空间管理与智能搜索，帮助用户实现从内容生产到传播的完整闭环。平台集成文生文、文生图以及智能PPT等AI驱动创作工具，提升了创作者在多模态内容生产中的效率与灵感表达。

#### 个人职责

1. **负责数据同步模块设计与实现**
   - 基于XXL-JOB+Redis+RocketMQ+OpenFeign构建ElasticSearch与MySQL的数据同步方案
   - 实现全量同步+增量同步机制，每15分钟执行一次定时任务
   - 通过批量操作和递归重试机制确保同步成功率，失败操作记录到文件并邮件通知
   - 使用HashSet优化修改操作，避免对已删除数据的无效同步

2. **负责大文件上传优化**
   - 实现前端分片上传与断点续传逻辑，支持多线程并发上传（弱网3-4线程，强网5-6线程）
   - 后端使用Map记录已上传分片索引，配合MinIO实现分片存储与合并
   - 解决了大视频上传速度慢和网络波动导致的上传失败问题
   - 设计了分片哈希校验机制，确保上传过程中数据完整性

3. **负责大模型接入与多账号管理**
   - 集成讯飞星火大模型（文生文、文生图、智能PPT）
   - 设计账号凭证池突破QPS=2的限制，通过使用人数标记实现多账号并发调用
   - 实现文生文的WebSocket流式输出，文生图和文生PPT的HTTP请求响应
   - 支持上下文历史记录管理，提升对话连贯性

4. **负责用户认证与安全机制**
   - 基于Spring Security+双JWT实现无感刷新token（短token 30min用于鉴权，长token 7天用于刷新）
   - 在网关层统一鉴权与授权，实现单点登录
   - 设计设备指纹和IP地址校验机制，防止token被盗刷
   - Token存储在HttpOnly Cookie中，提升安全性

5. **参与实时通信功能开发**
   - 基于WebSocket服务端中转架构实现一对一实时私聊功能
   - 使用HashMap管理用户WebSocket连接，实现消息转发
   - 设计心跳机制和异常连接清理机制，保证连接稳定性
   - 所有聊天消息持久化到MySQL，支持离线消息查询

6. **负责网关层开发优化**
   - 利用Gateway+Redis+XXL-JOB实现QPS统计和动态路由切换
   - 使用CompletableFuture+LinkedBlockingQueue实现请求合并处理
   - 定时任务每500ms批量处理队列中的请求，降低数据库压力
   - 设计临界值测试方案，动态判断是否启用合并请求机制

#### 项目难点

1. **分布式数据一致性问题**
   - 挑战：高并发场景下MySQL与ES的数据同步延迟，以及同步失败导致的数据不一致
   - 解决方案：
     - 采用全量同步（首次）+增量同步（定时）的混合策略
     - 通过RocketMQ缓存增量消息到Redis，按"新增→删除→修改"顺序批量同步
     - 设计递归重试机制（最多10次），失败操作记录文件并邮件通知
     - 使用HashSet记录文档ID，过滤对已删除数据的无效修改操作
   - 效果：保证了最终一致性，同步成功率达99%以上

2. **大文件上传稳定性问题**
   - 挑战：大视频文件（GB级）上传耗时长，网络波动易导致上传失败，用户体验差
   - 解决方案：
     - 前端将文件分片（每片5-10MB），多线程并发上传，充分利用带宽
     - 后端Map记录已上传分片索引，支持断点续传，避免重复上传
     - 上传前查询分片状态，已上传的分片直接跳过
     - 所有分片上传完成后，MinIO服务端执行分片合并
   - 效果：弱网环境下上传成功率提升60%，上传速度提升3-5倍

3. **大模型QPS瓶颈问题**
   - 挑战：讯飞星火单账号QPS=2，无法满足多用户并发调用需求
   - 解决方案：
     - 设计账号凭证池，每个账号维护当前使用人数（初始为0）
     - 用户请求时遍历凭证池，选择使用人数<2的账号，使用人数+1
     - 使用完毕后使用人数-1，释放凭证供其他用户使用
     - 类似Redis分布式锁的获取与释放机制
   - 效果：N个账号支持2N用户并发调用，QPS线性扩展

4. **Token安全性与用户体验平衡问题**
   - 挑战：Token过期时间短影响用户体验，过期时间长增加安全风险
   - 解决方案：
     - 采用双Token机制：短Token（30min）用于鉴权，长Token（7天）用于刷新
     - 前端每25分钟自动刷新短Token，用户无感知
     - 增加设备指纹和IP地址校验，防止Token被盗刷
     - Token存储在HttpOnly Cookie中，防止XSS攻击
   - 效果：在保证安全性的前提下，用户7天内无需重新登录

### 项目二：元域灯塔
**时间**: 2024.10 - 2024.12

#### 项目描述
本项目是一个捕捉、复制与重组现代社交文化基因的实验室。聚合剧本杀、桌游等最具传染性的娱乐模因。用户通过身份绑定，体验高热模因元隆、组队指令、文化表达、模因传播链及地理熔增探测，参与社交文化的演化。

#### 个人职责

1. **实现秒杀场景下的库存精准扣减**
   - 采用乐观锁机制（版本号/时间戳）实现库存扣减，解决超卖问题
   - 高并发下优先查询Redis库存进行预扣减，通过消息队列异步扣减MySQL
   - 设计补偿机制：MySQL扣减失败时反向恢复Redis库存，确保最终一致性
   - 定时任务校验Redis与MySQL库存差异，兜底修正数据

2. **设计Redis缓存防护体系**
   - 使用逻辑过期策略防止缓存击穿：缓存永不过期，后台异步更新
   - 缓存穿透防护：缓存空对象，设置较短过期时间
   - 缓存雪崩防护：设置随机过期时间，避免大量缓存同时失效
   - 有效降低数据库压力，保障系统稳定性

3. **实现Redis+Lua脚本的原子化下单预检**
   - Lua脚本原子性执行"查库存→查限购→扣库存→记次数"操作
   - 避免高并发下多步操作被插队导致的数据不一致
   - 减少网络往返次数，单次请求完成所有校验和预扣减
   - 配合RabbitMQ异步处理订单，实现系统解耦和流量削峰

4. **搭建Caffeine+Redis二级缓存架构**
   - 一级缓存（Caffeine本地缓存）：优先查询，命中直接返回
   - 二级缓存（Redis分布式缓存）：一级未命中时查询，并回写一级缓存
   - 更新策略：先更新数据库，再删除两级缓存（避免并发更新问题）
   - 显著提升热点数据访问速度，降低Redis和数据库压力
